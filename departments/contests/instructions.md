# コンテスト発見部門

## 目的
毎朝、データサイエンス・機械学習系のコンテストプラットフォームを横断して調査し、新規に見つかった/開始された募集中コンペを、賞金情報・AIツール利用可否とともに一覧化する。まだどのコンペにも参加していない段階の「発見・通知」が役割であり、進捗管理は行わない。

## 対象プラットフォーム
- Kaggle
- DrivenData

上記2つに絞っている理由: 2026-07-05の試験実行で、SIGNATE・Zindi・HackerEarthはサイトがJS主体のSPAでWebFetchでは中身が取得できず、検索スニペットも古い(締切済みを「募集中」と誤認しやすい)ことが判明したため対象外とした。AIcrowdは一部成功したが安定しないため今回は見送り。Nishikaは2026年3月末サービス終了、atmaCupも直近開催なしを確認済み。

将来的に対象を広げる場合は、まず該当サイトが静的ページで情報取得できるかを軽く試してから追加すること。

## 調査方法
- 各プラットフォームの公式コンペ一覧ページや信頼できる紹介記事をWeb検索/WebFetchで確認する
- 「募集中(応募締切前)」のコンペを対象とする。終了済み・締切済みのものは対象外
- 前日以前のレポート(`departments/contests/reports/` 内の直近ファイル)があれば読み、そこに無い新しいコンペを優先的に取り上げる。新規コンペが見つからないプラットフォームは、その日は取り上げなくてよい(毎回全プラットフォームを網羅する必要はない)
- 賞金・賞品の有無と金額を確認できる範囲で記載する。分からない場合は「賞金情報不明」と明記する(無理に断定しない)
- 締切日が分かれば記載する
- コンペの概要(何のタスクか、どんなデータを使うか、何を予測・分類・生成するか)を1文程度で記載する
- **各コンペのルール(Rules/規約ページ)を確認し、Claude Code等のAIコーディング支援ツール・LLMの利用可否を調べる。**判断は以下の目安に従う
  - ルール上、外部AIツール・自動化ツール・LLMの使用を明確に禁止していない → 「AI利用: 許可(明記なし)」
  - コード生成AI・LLMの利用を明示的に許可している → 「AI利用: 許可」
  - 「手動でのコーディングのみ」「AI支援禁止」等の明記がある → 「AI利用: 禁止」
  - ルールページを確認しても判断できない → 「AI利用: 要確認」
- 同一コンペの重複掲載を避ける

## 出力先
`departments/contests/reports/YYYY-MM/YYYY-MM-DD.md`(月ごとにフォルダを分け、その中に1日1ファイル)

## 出力フォーマット
- プラットフォームごとに見出しを分け、その中で通し番号(ファイル全体を通して連番)付きの箇条書きにする
- 1コンペにつき、概要1文+条件行(賞金/締切/AI利用)の2行程度。詳しい背景説明までは不要(シンプル優先)
- 新規コンペが無かったプラットフォームの見出しは省略してよい

```md
# コンテスト発見レポート - YYYY-MM-DD

## Kaggle
1. [コンペ名](URL)
   衛星画像から森林減少エリアを検出する画像セグメンテーションタスク。
   賞金: $XX,XXX / 締切: YYYY-MM-DD / AI利用: 許可
2. [コンペ名](URL)
   店舗の売上データから来月の需要を予測する回帰タスク。
   賞金情報不明 / 締切: YYYY-MM-DD / AI利用: 要確認

## DrivenData
3. [コンペ名](URL)
   衛星データから災害被害エリアを推定するタスク。
   賞金: $XX,XXX / 締切: YYYY-MM-DD / AI利用: 許可(明記なし)
```

新規コンペが1件も見つからない場合:
```md
# コンテスト発見レポート - YYYY-MM-DD

本日は新規の募集中コンペなし。
```

## 優先度・件数の目安
- 件数の上限は設けない。無理に数を揃えず、実際に見つかった新規コンペのみ記載する
- AI利用が「禁止」のコンペも、後で参加除外の判断材料になるため記載自体は行う(除外はしない)
